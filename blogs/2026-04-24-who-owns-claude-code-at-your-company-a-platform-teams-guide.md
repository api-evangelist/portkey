The context is yours to build.</em></p><p>Coding agents have become part of the default developer stack. <a href="https://www.anthropic.com/claude-code?ref=portkey.ai">Claude Code</a>, <a href="https://www.cursor.com/?ref=portkey.ai">Cursor</a>, and <a href="https://openai.com/codex?ref=portkey.ai">Codex</a> have gone from experimental to expected over the past year. A developer can install one, connect an API key, and be productive within minutes. The individual experience is no longer the constraint.</p><p>What varies, often dramatically, is the quality of output across a team using the same tools. That difference is not explained by the model. It is explained by the context.</p><p>Model labs and agent platforms understand this. Anthropic ships a <a href="https://portkey.ai/docs/guides/coding-agents/skills?ref=portkey.ai">skills system</a> and a native MCP client. Cursor adds skills and MCP integration. Codex follows the same pattern. The harness is converging. The context is what separates teams.</p><h2 id="the-engineers-who-get-more-out-of-agents">The engineers who get more out of agents</h2><p>In every engineering organization, a small number of developers push these tools further than everyone else. They encode review standards into reusable <a href="https://agentskills.io/?ref=portkey.ai">SKILL.md files</a>. They structure debugging workflows. They connect agents to internal systems through <a href="https://portkey.ai/docs/product/mcp-gateway?ref=portkey.ai">MCP integrations</a> -- Linear, Sentry, internal APIs, deployment tools. Over time, they build a working layer of context around the model that makes it behave like a senior member of their specific team.</p><p>Their agents produce noticeably better output. More relevant code, better reviews, sharper analyses. Everyone else is working with defaults.</p><p>That context rarely spreads. It lives in local directories, personal configuration files, and undocumented workflows that never make it into a shared repository. The same agent, inside the same company, produces inconsistent results depending entirely on who set it up.</p><p>This is not a limitation of coding agents. It is a knowledge distribution problem, the same one that shows up with runbooks, internal tooling, and onboarding docs. The best knowledge exists somewhere. It just doesn&apos;t travel.</p><h2 id="what-breaks-as-teams-scale">What breaks as teams scale</h2><p>When a platform team finally gets asked to &quot;sort out the AI setup,&quot; what they inherit is usually a version of the same thing: dozens of individual configurations that evolved independently, credentials stored in ways that won&apos;t survive an offboarding, no shared baseline for how agents should behave, and no visibility into what is being spent or by whom.</p><p>The engineers using these tools are not doing anything wrong. They adopted quickly, configured thoughtfully, and got genuine value. The problem is that none of it was designed to be shared. Skills live in local directories. MCP configs live in JSON files with credentials baked in. The engineer who set something up well is the only one who knows it exists.</p><p>This is the standard condition before infrastructure catches up with adoption. It happened with cloud access, with secrets management, with observability tooling. The pattern is consistent: individual adoption outpaces organizational infrastructure, and then platform teams inherit the cleanup. The question is not whether this happens, but how long it takes to recognize it and how much is lost in the meantime.</p><h2 id="ubers-platform-team-has-already-solved-this">Uber&apos;s Platform team has already solved this</h2><p><a href="https://x.com/trq212?ref=portkey.ai">Uber&apos;s engineering team</a> built an internal CLI as the unified entry point for all their coding agents and added Skills and MCP configurations as modular plugins that teams could contribute to and reuse. Their internal skills library grew from two to more than five hundred entries in five months, with a marketplace where the most useful context got promoted org-wide.</p><p>Most teams will not operate at Uber&apos;s scale, but the architecture is general. Every team has one engineer who has built this privately. The question is whether it stays on their laptop or becomes infrastructure.</p><h2 id="a-unified-registry-for-skills-and-mcp">A unified registry for Skills and MCP</h2><p><a href="https://portkey.ai/?ref=portkey.ai">Portkey</a> gives platform teams a single control plane for everything an agent needs: model routing, MCP tools, and Skills, all governed, versioned, and distributed with one command.</p><h3 id="skills-registry">Skills Registry</h3><p>Portkey&apos;s <a href="https://portkey.ai/docs/guides/coding-agents/skills?ref=portkey.ai">Skills Registry</a> is a central place to author, version, review, and sync skills across every agent your team uses. Skills are authored in <a href="https://portkey.ai/docs/product/prompt-engineering-studio?ref=portkey.ai">Prompt Engineering Studio</a> as <a href="https://portkey.ai/docs/product/prompt-engineering-studio/prompt-partial?ref=portkey.ai">Prompt Partials</a> with YAML frontmatter:</p><pre><code class="language-yaml">---
name: code-reviewer
description: Use when reviewing code or PRs. Apply company review standards.
---

# Code Review Standards

Lead with the most important issue.
Explain why it matters, not just what it is.
Suggest a specific fix with code.
Group by severity: critical, warning, suggestion.
</code></pre><p>Each skill gets a stable ID, full version history, and a draft/publish workflow. An engineer authors a skill, saves it as a draft, a tech lead reviews and approves, and published skills are what syncs. Drafts stay invisible. If a published skill causes problems, roll back to the previous version in one click.</p><figure class="kg-card kg-image-card"><img alt="Who owns Claude Code at your company? A platform team&apos;s guide to managing coding agents at scale" class="kg-image" height="1048" src="https://storage.ghost.io/c/04/80/04808bfa-0485-4139-98cc-e3f82ab7f009/content/images/2026/04/image-6.png" width="1920" /></figure><p>Syncing to every agent is one command:</p><pre><code class="language-bash">npx portkey skills sync
</code></pre><p>The CLI pulls your published skills and writes them to the correct directory for each agent automatically -- <code>.claude/skills/</code> for Claude Code, <code>.cursor/skills/</code> for Cursor, <code>.codex/skills/</code> for Codex, <code>.github/skills/</code> for GitHub Copilot. When you update a skill and publish it, every developer gets the new version on their next sync without touching their own configuration.</p><h3 id="mcp-registry">MCP Registry</h3><p>Portkey&apos;s <a href="https://portkey.ai/docs/product/mcp-gateway?ref=portkey.ai">MCP Registry</a> solves the same distribution problem for tools. Add MCP servers to your Portkey workspace with centralized auth -- OAuth, client credentials, or static headers -- and the <a href="https://portkey.ai/docs/guides/coding-agents/agent-cli?ref=portkey.ai">Portkey CLI</a> writes the correct configuration to each engineer&apos;s agent on setup.</p><figure class="kg-card kg-image-card"><img alt="Who owns Claude Code at your company? A platform team&apos;s guide to managing coding agents at scale" class="kg-image" height="1044" src="https://storage.ghost.io/c/04/80/04808bfa-0485-4139-98cc-e3f82ab7f009/content/images/2026/04/image-5.png" width="1920" /></figure><p>No more passing credentials over Slack. No more broken MCP configs that work on one machine and fail on another. The platform team manages server access in Portkey. Engineers get the tools automatically when they run the CLI.</p><h3 id="portkey-cli">Portkey CLI</h3><pre><code class="language-bash">npx portkey setup
</code></pre><p>That is the full install. The wizard asks which agent you are setting up, validates your Portkey key, lets you pick a provider or routing config, adds your workspace&apos;s MCP servers, and syncs shared skills. What gets configured under the hood covers <a href="https://portkey.ai/docs/product/ai-gateway?ref=portkey.ai">gateway routing</a> so every request appears in <a href="https://portkey.ai/docs/product/observability?ref=portkey.ai">Portkey logs</a> with cost, latency, and full trace detail, <a href="https://portkey.ai/docs/product/enterprise-offering/budget-policies?ref=portkey.ai">hard budget limits</a> per developer or team, and <a href="https://portkey.ai/docs/product/ai-gateway/fallbacks?ref=portkey.ai">auto-failover</a> to Bedrock or Vertex if your primary provider goes down. Developers change nothing about how they use their agent. They run one command.</p><p>The analogy that keeps coming up internally: coding agents are Docker. Portkey is Kubernetes. Docker was a step change for individual developers. Kubernetes is what made it possible to run Docker at organizational scale &#x2013; centralized configuration, resource controls, observability, rollout management. You do not need Kubernetes for one container. You need it when you are running a fleet.</p><h2 id="the-platform-teams-role">The platform team&apos;s role</h2><p>Coding Agents do not maintain themselves. Someone needs to own the registries: deciding what gets published, reviewing contributions, deprecating what is outdated, and promoting what works.</p><p>This is platform work, the same kind of work that goes into CI pipelines, service templates, and infrastructure provisioning. The goal is not to restrict developers but to give them a strong and consistent starting point.</p><p>Start with five skills and your most-used MCP servers. Let engineers contribute skill drafts. Review and publish the ones that generalize well. After a quarter, the library reflects how your organization actually builds -- not how a generic agent thinks you do.</p><p>The gap between your best engineer&apos;s agent output and everyone else&apos;s is not the model. It is the context. A skills and MCP registry closes that gap org-wide.</p><h2 id="get-started">Get started</h2><pre><code class="language-bash">npx portkey setup
