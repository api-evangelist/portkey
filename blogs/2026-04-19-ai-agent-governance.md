<p>Most teams deploying agents in production are moving faster than their safeguards can keep up. Existing controls were designed for single LLM requests: one prompt in, one response, maybe a content filter at the boundary.&#xa0;</p><p>Agents don&apos;t work that way. They chain model calls, invoke tools, and trigger external side effects across multiple steps. Without runtime control, this leads to unauthorized tool access, runaway costs, and outputs you can&#x2019;t trace back to a clear decision path.</p><p>This needs to be fixed at the infrastructure level.</p><h2 id="why-existing-safeguards-break-down-for-ai-agents">Why existing safeguards break down for AI agents</h2><p>The control surface changes once you move from a single request to a multi-step execution chain.</p><p>Traditional safeguards operate at the request boundary. You validate input, inspect output, and apply rate limits per call. That model assumes you know exactly where decisions are made.&#xa0;</p><p>In an agent run, you don&apos;t.&#xa0; Each step introduces a new decision, often with side effects. The sequence matters as much as any individual step.</p><p>Even standard logging falls short. HTTP-level logs capture requests, but not the reasoning, tool selection, or intermediate state changes within a run.</p><p>Cost control breaks in the same way. Per-request limits don&apos;t help when an agent chains dozens of calls across a single task. Also, with agents costs can compound. Agents spawn sub-agents, failed tool calls retry, parallel branches fan out. A single user action can trigger hundreds of LLM calls you never explicitly authorized, and by the time the session ends, the damage is done.</p><h2 id="what-ai-agent-governance-actually-means">What AI Agent governance actually means</h2><p>Governance for agents comes down to controls that shape how an agent behaves across a full run. There are five distinct pillars, and they only work when treated as interdependent.</p><h3 id="execution-control">Execution control</h3><p>Agents run autonomously, which means decisions compound before anyone can intervene. Control during execution would mean defining where human oversight can be inserted, what triggers an interruption, and how the system responds when something goes off-course.</p><p>Without this, agents either run to completion unchecked or fail abruptly with no recovery path.</p><h3 id="tool-and-action-permissions">Tool and action permissions</h3><p>In traditional systems, you validate a fixed call path. You know in advance which APIs will be called, so you can set permissions at design time.</p><p>Agents don&apos;t work that way. The agent decides which tools to invoke at runtime, based on the task and context. </p><p>MCP makes this concrete. An MCP server exposes a catalog of tools to any agent that connects to it. But without explicit controls, a connected agent gets access to the full catalog, regardless of what the task actually requires. </p><p>Effective tool permissioning means controlling which tools an agent can even discover, not just which ones it can execute. It also means permissions need to be evaluated at the moment a tool is invoked, not just at the start of a session because an agent evolves its behavior throughout a session as it interprets new context, and can drift into unauthorized territory as its chain of thought progresses. </p><p>A central MCP tool registry makes this manageable at scale. Instead of each MCP server handling its own access logic, permissions are defined and enforced in one place and updated without touching AI Agent workflow.</p><h3 id="cost-and-resource-governance">Cost and resource governance</h3><p>A single agent session can chain dozens of LLM calls across multiple models and tools. No individual call looks expensive, the cost accumulates across the session. By the time the run completes, you&apos;ve already overspent.</p><p>What&apos;s needed is the ability to set a budget at the level where spending actually happens i.e. on the project, the agent, the team, and API keys and have it enforced before the run completes, not after. That means tracking cumulative spend across every call in a session, and stopping or alerting when a defined threshold is crossed.</p><h3 id="policy-enforcement-layer">Policy enforcement layer</h3><p>Most guardrail implementations sit at the LLM boundary.</p><p>For agents, this might not be enough. The decisions that carry real risk happen mid-run, when the agent is invoking tools, calling APIs, or reading and writing data. An agent connected to MCP servers can trigger dozens of tool calls between any two LLM responses. None of those calls pass through the LLM boundary, so boundary-level guardrails never see them.</p><p>Policy enforcement for agents means applying checks at every action point in the run. What data is being accessed, what action is being taken, and does it comply with the policies defined for this agent and this context.</p><p>Without this, guardrails give you a false sense of control. You&apos;re watching the door while the agent acts freely inside the house.</p><h2 id="enforcing-ai-agent-governance-at-the-infrastructure-layer">Enforcing AI Agent governance at the infrastructure layer</h2><p>Governance embedded inside individual agent frameworks is fragile. It varies by framework and rarely carries across environments. A more reliable approach is to enforce it in a shared layer between agents and the systems they call.</p><p>Portkey&apos;s <a href="https://portkey.ai/docs/product/ai-gateway?ref=portkey.ai" rel="noopener noreferrer nofollow"><u>AI Gateway</u></a> applies controls directly in that request path. For MCP tools, access is provisioned centrally, so agents can only reach approved systems.</p><p>All activity is captured as a single run trace, covering model calls, tool usage, and cost. This gives teams a clear view of what happened without relying on scattered logs.</p><p>Because these controls sit outside the agent, they apply consistently without requiring changes to agent code.</p><p>A centralized gateway brings these controls together in one place, making them consistent across agents and environments.
